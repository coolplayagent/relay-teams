## 流程原则（必读）

**本技能采用严格的两阶段五步流程**：

1. **阶段 1：模板识别**：检查用户是否指定了模板风格（如华为风格）
2. **阶段 2：输入验证**：检查 `ppt_plan.md` 是否存在 → **如不存在则报错终止**
3. **阶段 3：图像准备**：搜索并准备页面所需的图像素材
4. **阶段 4：演示文稿生成**（两阶段）：
   - **阶段 4.1（Draft）**：生成低分辨率 HTML（640×360），验证布局和内容完整性
   - **阶段 4.2（成品）**：基于 draft 生成高分辨率 HTML（1280×720），精装修视觉
5. **阶段 5：PPTX 交付**（必选）：将成品 HTML 转换为 PPTX
   - 最终输出为 `{output_dir}/pages.pptx`（draft 和成品 HTML 均保留）

**⚠️ 关键流程警告**：

- **必须提供** `ppt_plan.md` 文件作为输入，否则报错终止
- **禁止**在无 `ppt_plan.md` 时尝试生成或推测内容
- 所有生成工作必须严格依据 `ppt_plan.md` 执行，不得偏离
- **阶段 4.1 全部完成前，不得启动阶段 4.2**

**两阶段设计理念**：

- 阶段 4.1（Draft）：快速生成内容完整的低分辨率稿，确保信息不遗漏
- 阶段 4.2（成品）：在 draft 基础上精装修，坐标放大两倍、视觉精细化
- draft 文件全程作为参考基准，最终 PPTX 由成品 HTML 转换生成

**输入文件要求**：

- 文件路径：工作目录下的 `ppt_plan.md`
- 格式要求：必须包含「大纲总览」和「页面详细描述」章节

### 阶段 4：演示文稿生成

本阶段采用两阶段隔离执行方式：先完成所有页面的阶段 4.1（draft），再启动阶段 4.2（成品）。

#### 4.1 生成低分辨率 Draft HTML

**目标**：在低分辨率（640×360）下生成内容完整、布局合理、可独立阅读的页面。不是骨架占位，而是"缩小版的成品"。

**规格**：
- 页面容器：`.ppt-slide { width: 640px; height: 360px; padding: 20px; box-sizing: border-box; overflow: hidden; }`
- 内容区域：600×320px（左右 20px 边距，上下 20px 边距）

**内容要求**：
- 所有文字必须是真实内容，禁止使用占位矩形代替文字
- ppt_plan.md 中该页的全部信息点都必须体现，不遗漏
- 标题、正文、表格数据、标签文字等全部填入真实文本
- 数据/对比/趋势 → 使用 ECharts/Chart.js 绘制实际图表
- 步骤/流程 → 绘制完整节点+连线+文字标注
- 并列信息 → 分栏/卡片化并填入内容

**布局要求**：
- 明确划分标题区、主体区、辅助区、结论区，版面饱满均匀
- 文字不超界、不重叠，重要模块尺寸大于次级模块
- 遵循风格参考文档的配色、字体、布局结构

**验收标准**：
- 页面在 640×360 尺寸下内容完整可读
- 不存在空白占位块或无文字的纯色矩形（装饰元素除外）
- 即使不经过阶段 4.2，也能传达完整信息

**文件命名**：`{output_dir}/page-N-draft.pptx.html`

#### 4.2 生成高分辨率成品 HTML

**目标**：在阶段 4.1 的完整内容基础上做精装修——视觉升级、装饰增强、局部内容深化。不推倒重来。

**规格**：
- 页面容器：`.ppt-slide { width: 1280px; height: 720px; padding: 40px; box-sizing: border-box; overflow: hidden; }`
- 内容区域：1200×640px（左右 40px 边距，上下 40px 边距）

**执行方式**：
- 使用独立 subagent（Agent 工具）逐页处理，每页单独调用
- subagent prompt 必传材料：
  a. 用户原始任务原文
  b. draft 文件路径（阶段 4.1 产物）
  c. 该页 ppt_plan.md 内容（大纲素材）
  d. 风格参考文档路径（如华为风格：`styles/huawei.md`）
  e. 阶段二规则（本文档路径）
  f. 输出文件路径
- subagent prompt **禁止**包含阶段 4.1 的生成指令或思考过程

**执行步骤**：
1. 用 Read 工具读取 draft HTML 文件（禁止凭记忆）
2. 阅读 prompt 中的大纲和风格素材，识别可补充的细节信息
3. 将 draft 坐标 ×2 作为基准定位，在模块内部做精装修

**布局约束**：
- 分区方案原样保留，坐标 ×2 基准，允许 ±20% 微调
- 元素只增不减，禁止引入新的大面积留白
- 空间变化就近消化

**精装修要求**（在模块内部完成，不改整体布局）：
- 视觉精细化：三级字体体系（标题 36-48px、副标题 24-28px、正文 16-20px）
- 线条精细化：精细线条（0.5~1px）
- 装饰增强：页面边缘/背景层加轻量几何装饰，不挤占内容空间
- 图表强化：简单图形升级为精致图表（如加网格线、数据标签、图例）
- 局部内容深化：关键数字加说明注释、结论加摘要高亮、列表拆为精致卡片

**禁止**：
- 重划分区、大幅移动模块、删除/合并模块、打散布局
- 降低信息密度

**验收标准**：
- 布局与 draft 一致，信息密度只增不减
- 模块内部有实质性视觉升级与装饰增强
- 达到"可直接用于正式展示"的完成度

**文件命名**：`{output_dir}/page-N.pptx.html`

### 阶段 4.3：布局 QA（强烈建议在转 PPTX 前执行）

在生成所有成品 HTML 页面后，先执行布局 QA，再进行 PPTX 转换。

1. **运行 QA 脚本**：
   ```bash
   node designer/slide_layout_qa.js {output_dir}
   ```
   - `{output_dir}` 必须是**包含 `.pptx.html` 文件的目录**
   - 可选配置文件命令：
     ```bash
     node designer/slide_layout_qa.js --config designer/slide_layout_qa.config.jsonc {output_dir}
     ```
   - 如只看终端报告、不落 JSON，可使用：
     ```bash
     node designer/slide_layout_qa.js --no-json {output_dir}
     ```

2. **读取 QA 结果**：
   - 终端会输出每一页的 `PASS` / `FAIL`
   - 默认会生成 `layout_qa_result.json`
   - 当 `{output_dir}` 为 `{session_dir}/pages` 时，JSON 输出路径为 `{session_dir}/layout_qa_result.json`

3. **阻断条件**：
   - 命令退出码非 `0`
   - `layout_qa_result.json` 中 `passed` 为 `false`
   - 存在 `failed_pages`

4. **建议检查清单**：
   - `blank_rate`：内容页空白率过高
   - `overflow`：元素超出页面边界
   - `text_occlusion`：文字被其他元素遮挡
   - `text_overlap`：文字之间重叠
   - `child_overflow` / `ancestor_clip`：子元素溢出或被祖先裁切
   - `block_overlap`：块级区域互相重叠
   - `recursive_blank`：局部区域出现明显空白块

### PPTX 转换兼容性边界（重要）

`html-to-pptx` 转换器对**常见排版 CSS**支持较好，但它**不是浏览器级 1:1 渲染引擎**。它的实际流程是：先用 Playwright 在真实浏览器里加载 HTML、执行脚本、应用样式，再读取浏览器的计算结果并映射到 PPTX。

这意味着：
- 不是“只支持内联样式”
- 也不是“Tailwind CSS 不支持”
- 内联样式、普通 class、Tailwind 类名、外链 CSS，只要在浏览器里真正生效，转换器通常都能读到
- 真正的边界在于：这些**已计算样式**能否被 PPT 里的文本、形状和图片稳定表达

**稳定支持，适合作为默认方案**：
- 普通文本、标题、段落、列表、基础表格
- 常规块布局、绝对定位、基础层叠
- `flex` / `grid` 的常见排版用法
- 纯色背景、常规边框、圆角、基础透明度
- 单层或常规的 `linear-gradient(...)` 背景
- 常规 `box-shadow`
- 常规图片、SVG、canvas 图表
- 少量 2D 旋转

**支持，但可能退化或部分栅格化**：
- 带透明通道的线性渐变
- `filter: blur(...)` 或 `backdrop-filter: blur(...)` 这类模糊效果
- 某些带裁切关系的圆角块、局部背景纹理、多层渐变背景
- 图标字体、`::before` / `::after` / `::marker` 承载的简单图标或标记
- SVG、图表、复杂装饰块在部分场景下会转成图片而不是可编辑形状
- 浏览器里依赖精细测量的文本换行，PPT 中可能出现轻微偏差
- 带 `rowspan` / `colspan` 的复杂表格，尤其是首行合并较多时

**当前不建议把核心信息建立在这些能力上**：
- 动画、过渡、hover/active、依赖运行时状态的交互效果
- 浏览器视口相关布局，如 `position: fixed`、`position: sticky`
- 复杂 `transform`，尤其是 `skew`、3D、`perspective`、连续矩阵变换
- `mix-blend-mode`、`mask`、复杂 `clip-path`
- 复杂 `filter` / `backdrop-filter` 组合，而不只是单纯的 `blur(...)`
- `radial-gradient(...)`、`conic-gradient(...)`、重复渐变、多层叠加渐变等高级背景效果
- 大量依赖伪元素承载正文、关键数字、关键标签的写法

**可能出现的退化方式**：
- 效果丢失或被简化
- 位置偏移
- 文字换行与浏览器不一致
- 元素被栅格化为图片
- 阴影、模糊、渐变、伪元素效果与原稿不一致

**推荐设计原则**：
- 核心信息优先放在普通文本、基础布局、基础图形上
- 复杂 CSS 可以用，但更适合做增强视觉，不适合作为唯一的信息承载方式
- 如果某个视觉模块严重依赖高级 CSS，优先先做一次转换验证；必要时将该模块预渲染为图片再参与转换

**失败预防建议**：
- 先生成结构稳定的 HTML，再做轻量视觉增强
- 高风险视觉效果优先作为装饰层，不作为唯一的信息承载主体
- 转 PPTX 前必须运行 QA，并人工抽查关键页面
- 对高风险页面，优先做“浏览器截图对照 + PPTX 结果复核”

#### 生成时的布局质量预检要求

##### 阶段 4.1 (Draft HTML) 布局要求

Draft 阶段（640×360）采用简化版布局要求，聚焦内容完整性：

**固定尺寸约束**：

```css
.ppt-slide {
  width: 640px;       /* 固定宽度 */
  height: 360px;      /* 固定高度 */
  overflow: hidden;    /* 超出隐藏 */
  padding: 20px;       /* 内边距保护 */
  box-sizing: border-box;
}
```

- 内容区域边界：左右 20px 边距，上下 20px 边距，有效区域 600×320px
- 所有元素必须严格限制在 640×360 边界内
- 预留 3px 安全边际

**字号约束**（Draft 阶段）：
- 标题：18-24px
- 副标题：12-14px
- 正文：8-10px
- 注释：6-7px

**内容密度**：Draft 阶段可适当简化装饰，确保内容完整可读即可。

##### 阶段 4.2 (成品 HTML) 布局要求

成品阶段（1280×720）采用完整版布局要求，聚焦视觉精细化：

**固定尺寸约束**

所有内容必须在固定容器内，禁止超出边界：

```css
.ppt-slide {
  width: 1280px;      /* 固定宽度 */
  height: 720px;      /* 固定高度 */
  overflow: hidden;    /* 超出隐藏 */
  padding: 40px;       /* 内边距保护 */
  box-sizing: border-box;
}
```

- 内容区域边界：左右 40px 边距，上下 40px 边距，有效区域 1200×640px
- 所有元素必须严格限制在 1280×720 边界内
- 预留 5px 安全边际，避免亚像素渲染导致的溢出

**2. 空白率控制（内容页 < 30%）**

- 内容页（type="content"）的视觉元素投影面积占比必须超过 70%
- 封面页、章节页、结束页不受此限制
- 避免大面积无内容的留白区域
- 通过增加图表、数据可视化、信息卡片等方式填充内容
- 排除项不计入空白：全屏背景、窄色条装饰线、低透明度元素、纯装饰圆圈、纯布局容器

**3. 内容密度控制**

- 单页信息量：每页控制在 3-5 个核心要点，避免信息过载
- 图文结合，优先使用代码生成数据可视化图表
- 避免大段文字堆砌，单页超过 200 字需分段或使用列表
- 保持页面 20-30% 的留白空间，让内容有"呼吸感"

**4. 溢出预防**

- 使用 `overflow: hidden` 或精确计算元素位置/尺寸防止溢出
- 文本内容预留足够空间，避免文字被截断
- 响应式字体缩放：标题 28-36px，副标题 20-24px，正文 14-18px，注释 12-14px
- 长文本分段或拆分到多页，文本超出时显示省略号

**5. 文本重叠避免**

- 绝对定位元素之间保持足够间距
- 文本元素之间的最小间距建议 16px 以上
- 检查相邻文本块的 bounding box 是否有交集风险
- 使用 flex/grid 布局时确保元素不会挤压重叠

**6. 元素遮挡避免**

- 合理设置 z-index 层级，确保文本始终在最上层可读
- 背景图/装饰元素使用较低 z-index 或作为背景层
- 卡片、图表等容器内的文本必须清晰可见，不被父元素或其他元素遮挡
- 使用半透明遮罩时确保不影响文字可读性

**7. 层级管理（Z-Index）规范**

```
背景层：z-index: 0
装饰层：z-index: 5
内容层：z-index: 10
遮罩层：z-index: 20
文本层：z-index: 50（始终在最上层）
```

**8. 布局策略**

**优先使用弹性布局**：

```html
<!-- ✅ 推荐：Flexbox 自动分配空间 -->
<div class="flex gap-6">
  <div class="flex-1">左侧内容</div>
  <div class="flex-1">右侧内容</div>
</div>

<!-- ✅ 推荐：Grid 网格布局 -->
<div class="grid grid-cols-2 gap-6">
  <div>第一列</div>
  <div>第二列</div>
</div>

<!-- ✅ 推荐：三列布局 -->
<div class="grid grid-cols-3 gap-4">
  <div class="col-span-1">窄列</div>
  <div class="col-span-2">宽列</div>
</div>

<!-- ❌ 避免：绝对定位导致溢出风险 -->
<div class="relative">
  <div class="absolute" style="left: 100px; top: 50px;">内容</div>
</div>
```

**9. 多卡片信息页防遮挡规则（新增）**

- 当页面包含"顶部导语/摘要框 + 下方三列卡片 + 底部页脚"时，导语区优先使用 `min-height`，避免标题换行后挤压正文。
- 这类三列信息页建议使用更保守的字号：
  - 导语标题：`26-28px`
  - 导语正文：`16px`
  - 卡片标题：`20-22px`
  - 卡片列表：`16px`
- 若页面同时包含导语区和页脚，卡片主内容区高度建议控制在 `340-360px`，避免底部页脚压住列表最后一项。
- 优先调整顺序：
  1. 缩短文案；
  2. 下调字号 2-4px；
  3. 缩小卡片间距与 padding；
  4. 最后才增加容器高度。
- 底部页脚区应增加 `border-top` 和 `padding-top`，与主内容形成稳定视觉与空间分隔。
- 三列页禁止让卡片标题超过两行；若超过两行，应优先缩短标题而非继续压缩正文区。
- 生成后三列页必须重点检查：
  - 导语框是否因固定高度压住正文；
  - 卡片列表最后一项是否接触页脚；
  - 卡片之间是否因内容换行发生视觉拥挤。

**绝对定位使用条件**：
- 仅在装饰元素、背景图层使用
- 内容区域禁止使用绝对定位
- 必须配合 `position: relative` 的父容器

**12 列网格系统**：
- 使用 12 列或 24 列网格系统，元素对齐到网格线，保持整齐

### 防空白核心策略

#### 内容扩展技术

当内容不足以填满可用空间时，采用以下扩展策略：

| 扩展技术 | 适用场景 | 操作方法 |
|---------|---------|---------|
| 视觉化转换 | 文字描述过多 | 将关键数据转为图表、图标、示意图 |
| 数据补充 | 数据支撑不足 | 添加趋势线、对比柱状、占比图示 |
| 案例填充 | 概念空洞 | 添加真实案例/引用/行业示例 |
| 图标装饰 | 内容稀疏 | 添加相关图标、装饰线条、背景形状 |
| 引用增强 | 观点单薄 | 添加名人名言、数据来源、权威背书 |

#### 布局补偿技术

- **元素放大**：将核心元素（图标、数字、标题）放大至视觉重心平衡
- **留白利用**：用渐变背景、装饰线条、logo 填补空白区域
- **对称平衡**：左右分布不均时添加呼应元素（如装饰色块）
- **视觉引导**：添加箭头、引导线连接分散的元素

#### 模块化填充

| 空白程度 | 填充策略 |
|---------|---------|
| 轻度空白（<15%） | 添加图标装饰、背景线条 |
| 中度空白（15-25%） | 添加辅助图表、次要信息 |
| 重度空白（>25%） | 重新规划布局，考虑拆分页面 |

### 响应式布局原则

#### 弹性容器设计

- 使用相对单位（%、rem）而非固定像素值定义容器宽度
- 图片和图表设置 `max-width: 100%` 防止溢出
- 表格设置横向滚动或自动换行机制

#### 动态字号系统

| 元素类型 | 字号策略 |
|---------|---------|
| 主标题 | 固定 36-48px，确保层次感 |
| 副标题 | 主标题的 60-80%，形成梯度 |
| 正文 | 16-20px，保证可读性 |
| 辅助文字 | 12-14px，颜色淡化处理 |
| **响应式规则** | 容器宽度 < 400px 时，字号缩小 10-15% |

### 叙事逻辑结构

**问题驱动型**：背景 → 问题 → 分析 → 方案 → 效果（适用于商业提案）
**时间线型**：过去 → 现在 → 未来（适用于发展历程）
**金字塔型**：结论 → 论据 → 细节（适用于汇报总结）
**对比型**：现状 A→ 现状 B→ 对比分析 → 结论（适用于竞品分析）

### 执行流程（两阶段）

#### 阶段 4.1：Draft HTML 生成

1. **读取大纲**：读取 `ppt_plan.md` 获取页面规划
2. **逐页生成 Draft**：
   - 解析页面列表
   - 逐页构建 640×360 的 HTML

#### 阶段 4.2：成品 HTML 生成

4. **运行布局 QA**：
   - 执行 `node designer/slide_layout_qa.js {output_dir}`
   - 检查终端输出是否存在 `FAIL`
   - 检查 `path.dirname({output_dir})/layout_qa_result.json` 是否生成
   - 若存在错误项，先修复页面，再进入 PPTX 转换

## 常见错误与解决方案

### 布局相关问题

| 问题         | 原因                           | 解决方案                           |
| ------------ | ------------------------------ | ---------------------------------- |
| 字体加载失败 | 字体名称拼写错误               | 使用字体库中列出的确切名称         |
| 图表不显示   | 容器 ID 错误或脚本执行时机问题 | 使用立即执行函数 (IIFE) 包裹图表代码 |
| 样式不一致   | Tailwind 类名冲突              | 使用!important 或更具体的选择器    |
| 内容溢出     | 内容过多超出 540px 高度        | 精简内容或调整布局                 |
| 图片加载失败 | 图片 URL 无效                  | 使用可靠的图片源或 base64 编码     |
| 空白过多     | 内容不足或布局过于稀疏         | 增加图表、补充信息或紧凑布局       |
| 元素溢出边界 | 位置/尺寸计算错误              | 检查 Tailwind 类名和内联样式       |

### 布局错误避免清单

**绝对禁止**：

- ❌ 不要使用绝对定位放置大量内容
- ❌ 不要依赖浏览器默认样式
- ❌ 不要假设内容长度（预留扩展空间）
- ❌ 不要忽视响应式（不同屏幕尺寸）
- ❌ 不要对内容区使用 `overflow: hidden`
- ❌ 不要使用 `fixed height + overflow` 组合

**强烈建议**：

- ✅ 所有元素距离边缘 ≥ 20px
- ✅ 元素间距 ≥ 16px
- ✅ 最小字号 ≥ 12px
- ✅ 文字与背景对比度 ≥ 4.5:1
- ✅ 优先使用 flex/grid 布局
- ✅ 绝对定位需配合 `position: relative` 父容器
- ✅ 预留 10% 缓冲空间应对内容扩展

### 典型问题解决方案

| 问题类型 | 具体表现 | 解决方案 |
|---------|---------|---------|
| **文字太多溢出** | 标题/正文超出容器边界 | ① 提炼核心文字，删除修饰词 ② 拆分为多页 ③ 启用智能换行或缩小字号 |
| **内容太少空白** | 画面空洞、留白过多 | ① 添加辅助图表或数据 ② 放大核心元素 ③ 添加图标装饰 ④ 补充案例/引用 |
| **图文比例失衡** | 图太大压文字 / 图太小看不清 | ① 文字多则图缩小做配图 ② 图为主则文字做说明标签 ③ 保持 6:4 或 7:3 的图文占比 |
| **元素相互遮挡** | 背景遮住文字 / 弹窗遮住关键信息 | ① 提高文字层 z-index ② 添加半透明背景 ③ 调整元素堆叠顺序 |
| **间距不协调** | 元素挤成一团 / 分散零乱 | ① 相关元素收紧（≤ 50px）② 不相关元素拉开（≥ 50px）③ 使用网格对齐 |

## 设计哲学

### 核心理念

> **"每一像素都应有其存在的意义"**

每一像素都应服务于信息的传递与视觉的体验。没有无缘无故的留白，也没有毫无意义的装饰。

### 设计三原则

| 原则 | 含义 | 实践 |
|-----|------|------|
| **必要性** | 每个元素都必须有存在的理由 | 删除无法解释其用途的元素 |
| **目的性** | 每个元素都应有明确的功能 | 装饰元素必须辅助信息理解 |
| **经济性** | 用最少的元素达成最好的效果 | 避免过度设计，信息密度适当 |

### 留白的艺术

- 留白不是浪费，而是给内容"呼吸"的空间
- 核心内容周围的留白可以引导视线、突出重点
- 低密度页面（如封面）的留白是一种视觉语言
- 高密度页面（如数据页）的留白可以分隔信息层级

### 视觉层级法则

1. **第一眼**：应看到最核心的信息（标题/数据/关键词）
2. **第二眼**：应看到辅助说明（图表/图标/次要文字）
3. **第三眼**：应看到装饰元素（背景/线条/品牌元素）

每个页面都应该有清晰的视觉层级，让读者在 3 秒内抓住重点。

### 布局即叙事

- 布局不仅是摆放元素，更是组织信息、引导阅读
- 重要的内容放在视觉重心位置（画面中心偏上 1/3 处）
- 相关内容应靠近，不相关内容应有明确分隔
- 阅读顺序应符合自然习惯（左到右、上到下）
